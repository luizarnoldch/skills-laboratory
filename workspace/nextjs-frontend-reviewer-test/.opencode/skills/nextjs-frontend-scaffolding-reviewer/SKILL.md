# Next.js Frontend Scaffolding Reviewer Skill

Validates frontend component structures, field coverage, and UI patterns generated for feature boundaries. It maps out strict formatting criteria to eliminate structural drift and surface deterministic errors to parent orchestrators.

## Rules

1. **Read-only execution state.** Never write, update, create, or drop code architecture or component configurations under any circumstances. This asset serves purely as an evaluation baseline.
2. **Mechanical analysis.** Do not use qualitative evaluations or speculative reasoning to assess component structures. The verification loop depends exclusively on the output flags generated by this skill's validation script (`validate.sh`, invoked by its resolved absolute path — see Path Resolution).
3. **Field coverage enforcement.** Ensure components use actual schema fields, not just placeholder ID fields. List components must display custom fields; forms must include inputs for all user-editable fields.

---

## Target Feature Component Maps

All generated frontend components must reside inside modular domain features at `src/features/[entity]/`.

### Case Standardization Map
* **Directory paths & file prefixes:** camelCase (e.g., `product`, `legalCase`)
* **React Components & TypeScript Types:** PascalCase (e.g., `ProductList`, `ProductFormCreate`)

### Frontend Layout Architecture

```
src/app/
└── [entity]s/
    └── page.tsx                        ← Thin page wrapper

src/features/[entity]/
├── schemas/
│   └── [entity].schema.ts              ← Zod schema (source of truth for fields)
├── views/
│   └── [Entity]View.tsx                ← Wrapper with ErrorBoundary + Suspense
└── components/
    ├── [Entity]List/
    │   └── index.tsx                   ← List display + CRUD toggles
    ├── [Entity]FormCreate.tsx          ← Create form with field inputs
    └── [Entity]FormUpdate.tsx          ← Update form with field inputs + entity prop
```

## Path Resolution

This skill's validation script lives inside the skill folder itself, not at
any project's root — there is no `scripts/` directory in the Next.js project
being validated. Never invoke a bare `scripts/validate.sh`; resolve the
absolute path first:

1. Get the repo root: `git rev-parse --show-toplevel` (works from any cwd).
   If that fails (not a git repo), walk upward from the current directory
   until you find a `.opencode/` directory — its parent is the repo root.
2. The script is at
   `<repo-root>/.opencode/skills/nextjs-frontend-scaffolding-reviewer/scripts/validate.sh`.
   Always invoke it by that absolute path.
3. `<target-folder>` (below) must be the **absolute path** to the Next.js
   project root being validated. If a calling agent handed you this path
   already, use it exactly as given — do not recompute or default it (never
   `.`).

## Validation Script Specifications

Running verification logic requires invoking the script by its absolute path
(see Path Resolution above), with an absolute `<target-folder>`:

```bash
bash <repo-root>/.opencode/skills/nextjs-frontend-scaffolding-reviewer/scripts/validate.sh <target-folder> <Entity> --transport <trpc|api>
```

### Component Validation Checks

The script validates three critical components:

#### 1. List Component (`[Entity]List/index.tsx`)
Checks:
- **File exists** at `src/features/[entity]/components/[entity]List/index.tsx` (or `[Entity]List.tsx`)
- **Schema import** present: `from "../../schemas/[entity].schema"`
- **Hook usage** detected: `useSuspenseList[Entity]s` or `useList[Entity]s`
- **Delete hook** present: `useDelete[Entity]`
- **Field display**: Table must display schema fields beyond just `id`
  - Parse schema to extract field names (exclude `id`, `createdAt`, `updatedAt`, `deletedAt`)
  - Verify `<th>` headers include field names OR verify `<td>` cells reference `[entity].[field]`
  - Baseline: at least one custom field displayed
- **Form integration**: Imports and renders `[Entity]FormCreate` and `[Entity]FormUpdate`

Failure conditions:
- Missing file
- Missing schema/hook imports
- Table only shows `id` column (no custom fields)
- Template comments still present (e.g., `{/* Add <th> for each schema field */}`)

#### 2. Create Form Component (`[Entity]FormCreate.tsx`)
Checks:
- **File exists** at `src/features/[entity]/components/[Entity]FormCreate.tsx`
- **Hook import** present: `useCreate[Entity]`
- **Form usage** detected: extracts `form` from hook, uses `form.handleSubmit()`
- **Field coverage**: Form must include inputs for user-editable schema fields
  - Parse schema to extract fields (exclude `id`, `createdAt`, `updatedAt`, `deletedAt`)
  - Check for `form.Field` components with `name=` attributes matching schema fields
  - Baseline: at least one field beyond submit buttons
- **No placeholders**: Template comments removed (e.g., `{/* Add form.Field for each schema field */}`)

Failure conditions:
- Missing file
- Missing hook import
- Template comments present
- No form fields (only buttons)
- Missing required schema fields

#### 3. Update Form Component (`[Entity]FormUpdate.tsx`)
Checks:
- **File exists** at `src/features/[entity]/components/[Entity]FormUpdate.tsx`
- **Hook import** present: `useUpdate[Entity]`
- **Entity prop** present: component accepts `[entity]` prop with proper type
- **Form usage** detected: extracts `form` from hook
- **Field coverage**: Form must include inputs for editable schema fields (same as Create, plus `id` if needed)
- **No placeholders**: Template comments removed

Failure conditions:
- Missing file
- Missing hook import
- No entity prop with proper typing
- Template comments present
- Missing schema fields

### Schema Field Extraction

The validator parses `schemas/[entity].schema.ts` to extract field names:

1. Locate the main schema export: `export const [entity]Schema = z.object({...})`
2. Extract keys from the `z.object({...})` block
3. Filter out auto-generated fields: `id`, `createdAt`, `updatedAt`, `deletedAt`
4. Remaining fields = user-editable fields that must appear in components

### Script Execution Contracts

The underlying shell scripts scan physical paths and dump strict output flags matching this pattern:

```
COMPONENT:list:PASS
COMPONENT:form-create:FAIL:Template comment still present;No schema fields detected in form
COMPONENT:form-update:PASS
VERDICT:PASS|FAIL
```

* **Component Flags:** `COMPONENT:<component_name>:PASS` or `COMPONENT:<component_name>:FAIL:<semicolon_separated_reasons>`
* **Final Judgment Token:** `VERDICT:PASS` or `VERDICT:FAIL` (This token must terminate the standard output buffer).

## Output Report Formats

The agent must translate raw script lines into the correct markdown block schemas below.

### 1. Unified Success Schema (VERDICT: PASS)

```markdown
## Frontend Review: [Entity]

### Verdict: PASS

All frontend components conform to the architecture. Field coverage complete. Nothing to fix.
```

### 2. Unified Failure Schema (VERDICT: FAIL)

```markdown
## Frontend Review: [Entity]

### Verdict: FAIL

### Component Results
| Component | Status | Details |
| --- | --- | --- |
| [Entity]List | [✅ | ❌] | [Description or direct script error notes] |

| [Entity]FormUpdate | [✅ | ❌] | [Description or direct script error notes] |

### Required Fixes
- **File**: `src/features/[entity]/components/[component_file]`
  - **Issue**: [Insert raw failure payload string from script line here]
  - **How to fix**: Adjust the component to include proper schema fields and remove template placeholders.

```

## Error State & Operational Overrides

| Environmental Trigger | Handling Path |
| --- | --- |
| Target workspace parameter path cannot be physically mapped | Halt processing. Inject `VERDICT:FAIL:target directory not found: <path>` directly into the layout report. |
| Feature directory `src/features/[entity]/` is missing | Inject `VERDICT:FAIL:frontend directory missing` into the layout report. |
| Schema file missing | Inject `VERDICT:FAIL:schema file not found — cannot validate field coverage` into the report. |
| Component files missing | Script will flag each missing component individually with `COMPONENT:<name>:FAIL:file not found`. |
