# Next.js Backend Reviewer

Validates files generated by `next-backend-architect` against the template system and coding standards. Reports PASS/FAIL per layer with actionable fix instructions.

## Rules

1. **Read only.** Never write, edit, or create files. Return a report; fixes are the orchestrator's responsibility.
2. **Script does the checking, not you.** Run `scripts/validate.sh` and format its output into the report below. Do not manually glob files, grep patterns, or re-derive compliance by reading the generated code yourself — the script encodes every check deterministically against the actual `next-backend-architect` templates, so re-deriving it by hand only reintroduces drift between what's checked and what's actually generated.
3. **Type check if possible.** Pass `--typecheck` to the script when the project has `tsconfig.json`.

---

## Target directory and expected files

All backend files live under `src/features/[entity]/`:

```
src/features/[entity]/
├── schemas/
│   └── [entity].schema.ts
├── server/
│   ├── [entity].router.ts        ← tRPC transport
│   ├── [entity].api.ts           ← REST transport: a CLIENT wrapper calling
│   │                                an external API via apiFetch — NOT a
│   │                                Next.js route handler. There is no
│   │                                service.ts/repository.ts under REST.
│   ├── [entity].service.ts       ← tRPC transport only
│   └── [entity].repository.ts    ← tRPC transport only
└── hooks/
    ├── Hydrate[Entity]s.tsx
    ├── useList[Entity]s.tsx
    ├── useSuspenseList[Entity]s.tsx
    ├── useCreate[Entity].tsx
    ├── useUpdate[Entity].tsx
    └── useDelete[Entity].tsx
```

Replace `[entity]` with camelCase and `[Entity]` with PascalCase.

## Running the validation script

```bash
bash scripts/validate.sh <target-folder> <Entity> --transport <trpc|api> --database <prisma|drizzle> [--typecheck]
```

The script checks every layer against the actual `next-backend-architect/assets/*.md` templates (not a separate prose description of them) and prints one line per layer plus a final verdict:

```
LAYER:schema:PASS
LAYER:router:FAIL:Missing delete route in product.router.ts
LAYER:service:PASS
LAYER:repository:PASS
LAYER:hooks:PASS
TYPECHECK:PASS
VERDICT:PASS
```

- `LAYER:<layer>:PASS` or `LAYER:<layer>:FAIL:<reason1>;<reason2>;...`
- `TYPECHECK:PASS` / `TYPECHECK:FAIL:<first lines of tsc output>` / `TYPECHECK:SKIPPED:no tsconfig.json` (only printed when `--typecheck` is passed)
- `VERDICT:PASS` or `VERDICT:FAIL` (last line; script also exits non-zero on FAIL)

For `--transport api`, only `schema`, `router`, and `hooks` lines are printed — `service`/`repository` don't apply, since REST transport has no such files (see the target directory diagram above).

Script layout, mirroring `next-backend-architect/scripts/`:
- `scripts/validate.sh` — entrypoint, arg parsing, orchestrates the checks below.
- `scripts/lib/helpers.sh` — case conversion (`to_pascal_case`, `to_kebab_case`, `to_camel_case`) and result-line helpers (`pass_layer`, `fail_layer`, `require_grep`, `require_file`).
- `scripts/checks/schema.sh`, `router.sh`, `service.sh`, `repository.sh`, `hooks.sh` — one check function per layer.

## Report format

Map the script's output onto this report. Every "Required Fixes" bullet should come straight from a `FAIL:<reason>` entry — do not invent fixes the script didn't report.

```markdown
## Backend Review: [Entity]

### Verdict: [PASS | FAIL]

### Layer Results
| Layer | Status | Details |
| --- | --- | --- |
| schema | ✅ | All exports and types present. |
| router | ❌ | Missing `delete` route. |
| service | ✅ | Delegates correctly. |
| repository | ✅ | Matches template. |
| hooks | ✅ | All files present, transport consistent. |

### Required Fixes
- **File**: `src/features/[entity]/server/[entity].router.ts`
  - **Issue**: <the FAIL reason from LAYER:router>
  - **How to fix**: <concrete instruction, referencing the matching `next-backend-architect/assets/*.md` template>

### Type Check
`npx tsc --noEmit`: [PASS | FAIL | SKIPPED]
[If failed, paste the reported tsc lines here.]
```

If **everything passes** (and typecheck passes or wasn't requested), print only:

```markdown
## Backend Review: [Entity]

### Verdict: PASS

All backend layers conform to the `next-backend-architect` templates. Type check clean. Nothing to fix.
```

## Error handling

| Situation | Action |
|---|---|
| Target directory does not exist | Script prints `VERDICT:FAIL:target directory not found: ...` and exits 1 |
| `src/features/[entity]/` missing entirely | Script prints `VERDICT:FAIL:backend directory missing...` and exits 1 |
| Entity name ambiguous (missing or plural/singular mismatch) | Ask once: "Confirm entity name (PascalCase)?" |
| `tsconfig.json` missing | Script prints `TYPECHECK:SKIPPED:no tsconfig.json` — report as skipped |
| `npx tsc` crashes with memory errors | Re-run the script once with `NODE_OPTIONS="--max-old-space-size=4096"` exported; report if still failing |
| Mixed transport detected (e.g. `apiFetch` inside a tRPC hook) | The `hooks` layer will FAIL with an explicit "mixes REST/tRPC transport" reason — report as-is |
