# Next.js Backend Scaffolding Reviewer Skill

Validates codebases, directory scaffolding structures, and source architectures generated for feature boundaries. It maps out strict formatting criteria to eliminate structural drift and surface deterministic errors to parent orchestrators.

## Rules

1. **Read-only execution state.** Never write, update, create, or drop code architecture or schema configurations under any circumstances. This asset serves purely as an evaluation baseline.
2. **Mechanical analysis.** Do not use qualitative evaluations or speculative reasoning to assess file structures. The verification loop depends exclusively on the output flags generated by this skill's validation script (`validate.sh`, invoked by its resolved absolute path — see Path Resolution).
3. **Transport enforcement.** Ensure structural analysis maps exclusively to the requested transport channel (`trpc` vs `api`). If REST (`api`) transport is used, explicitly bypass verification steps for layers that do not exist in that pattern (such as services or database repositories).

---

## Target Feature Directory Maps

All generated infrastructure components must reside inside modular domain features at `src/features/[entity]/`.

### Case Standardization Map
* **Directory paths & file prefixes:** camelCase (e.g., `product`, `legalCase`)
* **React Hooks & TypeScript Typings:** PascalCase (e.g., `useListProducts`, `HydrateProducts`)

### 1. tRPC Transport Layout Architecture

```
src/features/[entity]/
├── schemas/
│   └── [entity].schema.ts              ← Zod validation and type schemas
├── server/
│   ├── [entity].router.ts              ← tRPC procedure endpoint declarations
│   ├── [entity].service.ts             ← Business domain layer execution logic
│   └── [entity].repository.ts          ← Active database connection or query maps
└── hooks/
├── Hydrate[Entity]s.tsx            ← Server-side hydration wrapper components
├── useList[Entity]s.tsx            ← Collection query hook
├── useSuspenseList[Entity]s.tsx    ← Suspense-compatible collection query hook
├── useCreate[Entity].tsx           ← Mutation hook for resource creation
├── useUpdate[Entity].tsx           ← Mutation hook for resource modifications
└── useDelete[Entity].tsx           ← Mutation hook for resource destruction
```

### 2. REST Transport Layout Architecture

```
src/features/[entity]/
├── schemas/
│   └── [entity].schema.ts              ← Zod validation and type schemas
├── server/
│   └── [entity].api.ts                 ← HTTP client routing layer utilizing apiFetch
└── hooks/
├── Hydrate[Entity]s.tsx            ← Server-side hydration wrapper components
├── useList[Entity]s.tsx            ← Collection query hook
├── useSuspenseList[Entity]s.tsx    ← Suspense-compatible collection query hook
├── useCreate[Entity].tsx           ← Mutation hook for resource creation
├── useUpdate[Entity].tsx           ← Mutation hook for resource modifications
└── useDelete[Entity].tsx           ← Mutation hook for resource destruction
```

*Note: Service and repository layers must not exist under pure REST environments.*

## Path Resolution

This skill's validation script lives inside the skill folder itself, not at
any project's root — there is no `scripts/` directory in the Next.js project
being validated. Never invoke a bare `scripts/validate.sh`; resolve the
absolute path first:

1. Get the repo root: `git rev-parse --show-toplevel` (works from any cwd).
   If that fails (not a git repo), walk upward from the current directory
   until you find a `.opencode/` directory — its parent is the repo root.
2. The script is at
   `<repo-root>/.opencode/skills/nextjs-backend-scaffolding-reviewer/scripts/validate.sh`.
   Always invoke it by that absolute path.
3. `<target-folder>` (below) must be the **absolute path** to the Next.js
   project root being validated. If a calling agent handed you this path
   already, use it exactly as given — do not recompute or default it (never
   `.`).

## Validation Script Specifications

Running verification logic requires invoking the script by its absolute path
(see Path Resolution above), with an absolute `<target-folder>`:

```bash
bash <repo-root>/.opencode/skills/nextjs-backend-scaffolding-reviewer/scripts/validate.sh <target-folder> <Entity> --transport <trpc|api> --database <prisma|drizzle> [--typecheck]
```

### Schema Custom Fields Validation

For Prisma schemas, the validator checks that entity-specific fields have been filled in beyond the template's base structure. Two conditions trigger a failure:

1. **Placeholder detected** — the `...` template stub is still present in the file, meaning custom fields were never filled in.
2. **Only base fields** — the `z.object({...})` block contains only `id`, `createdAt`, `updatedAt`, or `deletedAt` fields. At least one entity-specific field (derived from the Prisma model) must be present.

For Drizzle schemas, custom fields are auto-derived from the Drizzle table definition via `createSelectSchema`/`createInsertSchema`/`createUpdateSchema`, so explicit field-counting validation is not applied.

### Script Execution Contracts

The underlying shell scripts scan physical paths and dump strict output flags matching this pattern:

```
LAYER:schema:PASS
LAYER:router:FAIL:Missing delete route in product.router.ts
LAYER:service:PASS
LAYER:repository:PASS
LAYER:hooks:PASS
TYPECHECK:PASS
VERDICT:FAIL
```

* **Layer Flags:** `LAYER:<layer_name>:PASS` or `LAYER:<layer_name>:FAIL:<comma_separated_reasons>`
* **Typecheck Flags:** `TYPECHECK:PASS`, `TYPECHECK:FAIL:<compilation_errors>`, or `TYPECHECK:SKIPPED:no tsconfig.json`
* **Final Judgment Token:** `VERDICT:PASS` or `VERDICT:FAIL` (This token must terminate the standard output buffer).

## Output Report Formats

The agent must translate raw script lines into the correct markdown block schemas below.

### 1. Unified Success Schema (VERDICT: PASS)

```markdown
## Backend Review: [Entity]

### Verdict: PASS

All backend layers conform to the templates. Type check clean. Nothing to fix.
```

### 2. Unified Failure Schema (VERDICT: FAIL)

```markdown
## Backend Review: [Entity]

### Verdict: FAIL

### Layer Results
| Layer | Status | Details |
| --- | --- | --- |
| schema | [✅ | ❌] | [Description or direct script error notes] |
| router | [✅ | ❌] | [Description or direct script error notes] |
| service | [✅ | ❌] | [Description or direct script error notes] |
| repository | [✅ | ❌] | [Description or direct script error notes] |
| hooks | [✅ | ❌] | [Description or direct script error notes] |

### Required Fixes
- **File**: `src/features/[entity]/[target_file_path]`
  - **Issue**: [Insert raw failure payload string from script line here]
  - **How to fix**: Adjust the code blocks in this layer to line up with standard architecture blueprints.

### Type Check
`npx tsc --noEmit`: [PASS | FAIL | SKIPPED]
```

## Error State & Operational Overrides

| Environmental Trigger | Handling Path |
| --- | --- |
| Target workspace parameter path cannot be physically mapped | Halt processing. Inject `VERDICT:FAIL:target directory not found: <path>` directly into the layout report. |
| Feature directory `src/features/[entity]/` is missing | Inject `VERDICT:FAIL:backend directory missing` into the layout report. |
| Mixed structures detected (e.g., mixing tRPC and REST imports) | The script will dump a `mixes REST/tRPC transport` failure payload. Transcribe this violation directly to the report without attempting to balance or reconcile the code manually. |
